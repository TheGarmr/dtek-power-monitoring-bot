# DTEK Power Monitoring Bot

Сервіс для моніторингу фактичних відключень електроенергії за адресами через ДТЕК. Він не опитує всі адреси постійно: моніторинг запускається лише після сигналу від `ntfy`, який зазвичай приходить від `healthchecks.io`, коли певний хост перестав надсилати ping.

## Що потрібно

- акаунт `healthchecks.io`;
- `ntfy` topic, публічний або self-hosted;
- Telegram bot token і chat id;
- хости або ESP32-пристрої, які пінгують свої checks у `healthchecks.io`;
- `addresses.json` з alias-ами адрес, які мають збігатися з повідомленнями з `ntfy`.

## Загальна схема

1. Хост, який живиться від потрібної електромережі, регулярно пінгує свій check у `healthchecks.io`.
2. Якщо ping не приходить вчасно, `healthchecks.io` переводить check у стан `DOWN`.
3. `healthchecks.io` надсилає повідомлення в `ntfy`.
4. Бот слухає JSON-stream цього `ntfy` topic.
5. Якщо бот бачить повідомлення виду `Alias is DOWN`, він починає моніторити відповідну адресу.
6. Якщо пізніше приходить `Alias is UP`, бот завершує моніторинг цієї адреси.

`Alias` має бути ключем у `src/config/addresses.json` або у файлі, який монтується в Docker як `/app/src/config/addresses.json`.

У `healthchecks.io` потрібно налаштувати ntfy-повідомлення так, щоб у `title` або `message` був текст формату `Alias is DOWN` для початку моніторингу і `Alias is UP` для завершення.

Приклад формату адрес:

```json
{
  "Квартира": {
    "city": "м. Одеса",
    "street": "вул. Прикладна",
    "houses": ["10", "10А"]
  }
}
```

## Як відбувається моніторинг

Коли надходить `DOWN`, сервіс перевіряє, чи є такий alias у конфігурації. Якщо адреса вже моніториться, повторний `DOWN` ігнорується.

Далі бот відкриває сторінку відключень ДТЕК через Playwright, отримує CSRF token і робить AJAX-запит для міста та вулиці з конфігурації. Для адрес із кількома варіантами будинку бот шукає будинок, де є релевантна інформація про відключення. Якщо таких кілька, обирається запис із найранішим очікуваним часом відновлення.

Якщо ДТЕК уже має дані, у Telegram надсилається повідомлення з причиною, часом початку, очікуваним відновленням, часом оновлення на сайті ДТЕК і часом перевірки ботом. Якщо даних на сайті ще немає, бот надсилає коротке повідомлення, що відключення зафіксовано, але інформація ДТЕК поки відсутня.

Після цього адреса записується в `artifacts/state.json`, і cron продовжує перевірки:

- якщо інформації ДТЕК ще немає, перевірка йде частіше, за `CRON_INITIAL_INTERVAL`;
- коли інформація з'явилася, інтервал змінюється на `CRON_INTERVAL`;
- основне Telegram-повідомлення редагується, а не дублюється;
- якщо ДТЕК додав очікуваний час відновлення, бот надсилає окреме повідомлення;
- якщо очікуваний час відновлення змінився, бот також надсилає окреме повідомлення.

Коли приходить `UP`, бот редагує основне Telegram-повідомлення: додає час відновлення, час початку відключення і тривалість. Якщо під час моніторингу надсилалося окреме повідомлення про появу очікуваного часу відновлення, бот також намагається видалити його. Після цього адреса видаляється зі стану, і моніторинг для неї зупиняється.

## Порядок повідомлень

1. `DOWN` без даних ДТЕК: повідомлення про зафіксоване відключення та відсутність даних на сайті.
2. `DOWN` з даними ДТЕК: повідомлення з причиною і діапазоном відключення.
3. Під час моніторингу: редагування основного повідомлення після нових перевірок.
4. Окреме повідомлення, якщо з'явився графік або очікуваний час відновлення.
5. Окреме повідомлення, якщо час відновлення перенесено.
6. `UP`: фінальне редагування основного повідомлення про відновлення електропостачання і видалення окремого повідомлення про появу очікуваного часу, якщо воно було надіслане.

## Конфігурація

Основні змінні оточення:

```env
TELEGRAM_BOT_TOKEN=
TELEGRAM_CHAT_ID=
NTFY_URL=https://ntfy.sh
NTFY_TOPIC=power-monitoring
NTFY_AUTH_TOKEN=
CRON_INITIAL_INTERVAL=2
CRON_INTERVAL=10
PORT=3000
```

`NTFY_AUTH_TOKEN` потрібен тільки для приватного або self-hosted topic з авторизацією.

## Запуск

Сервісний режим:

```bash
npm run start:service
```

Або через Docker Compose:

```bash
docker compose up -d --build
```

HTTP health endpoint доступний на:

```text
GET /health
```

Він повертає статус сервісу, поточний час і кількість адрес, які зараз моніторяться.

## ESP32

У репозиторії є приклад `esp32-light-pinger.yaml` для ESPHome. Такий пристрій може регулярно пінгувати `healthchecks.io`; якщо живлення зникло і ping перестав приходити, `healthchecks.io` запускає весь подальший ланцюжок через `ntfy`.
